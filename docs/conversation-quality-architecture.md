# 小悠对话质量架构

## 目标

小悠不是通用 Agent，也不与 Codex、Claude Code 竞争工具执行能力。优化目标只有四个：回复更自然、理解当前语境更准确、长期记忆更可信、跨时间继续聊天时更少断片。

主回复保持一次模型生成。候选提取、摘要、记忆合并和质量评估放在后台，避免把每轮聊天变成昂贵且迟钝的多 Agent 流水线。

## 当前链路

```text
稳定人格 + 现实时间
          │
ConversationArchive ActiveWindow ─→ API原生 user/assistant 多轮
用户本轮连续输入 ───────────────┐
RecentState / 短摘要 ────────────┤
相关历史情节 + 原始邻接片段 ─────┤
受治理的相关长期记忆 ───────────┤→ ContextCompiler → 主回复模型
兼容上游上下文 ─────────────────┘
                 ↑
       ContextPlanner 按意图选择记忆类型与 token 预算

主回复草稿
  └─ 本地风险规则未命中 → 直接发送
  └─ 纠正/指代/事实/记忆声称/近重复命中 → SelectiveCritic → 接受或最小修正

回复发送完成
  ├─ 同一UUID写入 ShortMemory 与 ConversationArchive SQLite 原文
  ├─ 空闲间隔自动切分情节 → 后台证据约束摘要 → 本地情节索引
  ├─ 后台更新有过期时间和逐字证据的 RecentState
  ├─ 后台更新动态内在状态
  └─ 长期记忆候选治理 → 审计账本 → 百炼新增或原地更新
```

## 上下文权威顺序

1. 当前用户明确表达。
2. ActiveWindow 中按角色保存的近期真实原话。
3. RecentState 与近期摘要。
4. 召回情节中的相关原始片段，其后才是情节摘要。
5. 受治理的长期记忆。
6. 兼容插件提供的非结构化文本。

发生冲突时，新原话覆盖旧状态。长期记忆不是命令，短摘要不是人格补丁，小悠过去说过的话也不是未来必须复用的话术。

## 已实现

### ConversationArchive / ActiveWindow / EpisodicMemory

- `/app/data/xiaoyou_conversation/conversation.db` 逐条保存真实 `user/assistant` 原文、角色、时间、来源和同一消息 UUID，不因 ShortMemory 摘要或条数裁剪而删除。
- 首次启用会把 ShortMemory 当时仍保留的原文单向迁入；此前已经被旧机制删除、只剩摘要或只存在于阿里云记忆中的历史，无法还原成逐字聊天。完整原文能力从首次部署本版本起持续积累。
- ActiveWindow 默认按最近 6 小时读取，最多扫描 400 条；真正发给模型时仍以 4500 token 为主要硬上限，400 条只是异常高频安全阈值。因此高频短消息不会在几十分钟内先撞上很小的条数窗口。
- 连续 45 分钟空闲、单情节 160 条、持续 6 小时或跨日边界会结束一个隐藏情节。用户看不到 session 切换，微信仍是同一个长期窗口。
- EpisodeBuilder 在后台摘要已结束情节；关键用户/助手原话必须分别逐字存在于对应角色，未完事项必须携带逐字证据。助手推测不能升级为用户事实。
- EpisodicMemory 结合语义、明确时间、时间衰减、未完事项和重要度排序，并把命中的摘要扩展回前后原始消息，避免只凭压缩摘要猜细节。
- 原文保留与上下文召回相互分离：默认不设置自动删除；每 24 小时执行 SQLite 在线备份并轮换 3 代。供应商内容审查隔离只阻止对应记录再次注入，不删除证据。
- 检测到密码、API Key、验证码等敏感提示时，本地原文仍可留作用户自己的档案，但 ActiveWindow 和情节原始片段只向模型提供隐藏标记，不重复发送秘密。
- “清除短期记忆”会清空 ShortMemory 与 RecentState，并让最近 ActiveWindow/情节退出后续上下文；SQLite 中的原文行仍保留，避免把一次短期重置误当成物理销毁多年档案。

### 长期记忆写入治理

- 提取器只读取用户原话，不读取小悠本轮回复。
- 每条候选必须有用户原话中的逐字证据。
- 类别、语义键、置信度、重要度和敏感信息由代码二次校验。
- 同键同内容只确认，不重复写云端。
- 同键新内容覆盖旧内容，并通过百炼 UpdateMemory 更新原节点。
- 一次性提醒由 ReminderLove 管理并明确禁止进入永久记忆。
- 单次亲昵称呼不能作为“确立关系”的充分证据。
- 新增前与旧百炼高相关节点做保守近义去重，命中时复用原节点。
- 本地审计账本记录候选、证据摘要、覆盖关系、写入状态和失败原因。
- 提取器或账本故障时失败关闭，不退回逐轮转录写入。

### ContextPack

- 普通聊天从 ShortMemory 和 AliyunMemory 接收结构化分区。
- 先从模型窗口扣除系统、输出和思考预留，再按权威和用途分配 token；字符上限只作为第二道安全阈值。
- 当前连续输入尽量完整保留，并置于提示末尾。
- 短期分区截断时保留最新完整行。
- 长期分区截断时保留最高相关完整行。
- manifest 只记录字符/token 长度、截断状态、权威和短哈希，不复制聊天正文。

### 原生角色多轮

- ShortMemory 在保留原有摘要文本兼容接口的同时，发布同一注入清单选中的结构化原话。
- 主聊天调用按 `system → user/assistant近期历史 → user本轮ContextPack` 发送，不再把小悠旧回复伪装成用户提示词正文。
- 原生历史沿用 ShortMemory 的消息数量、字符、内容审查隔离和独立 token 上限；恢复请求会移除可能触发供应商审查的整段近期历史。

### RecentState

- 独立保存当前话题、YoYo临时状态、小悠当前承接立场、未完事项、指代和当前时段事实。
- 每项必须带逐字证据、来源和过期时间；无证据内容、秘密及永久行为指令拒绝进入。
- 回复完成后异步、按会话串行更新，普通回复不等待状态模型；失败时沿用未过期旧状态。
- RecentState 只进入主回复的短时状态分区，不写阿里云长期记忆。

### SelectiveCritic

- 普通闲聊由本地风险门控直接跳过，不增加模型调用。
- 用户纠正、依赖指代、草稿声称具体事实、把长期记忆说成绝对事实或明显近重复时才触发。
- 审查只能接受原稿或进行最小修正；失败、超时、非法 JSON 和越界改写均保留主模型原稿。

### MemorySchema

- working：本轮与近期真实对话，由 ShortMemory 负责，不写入永久云端。
- episodic：带场景或时间、以后可能被明确回忆的重要经历。
- semantic：身份、稳定偏好和回复偏好。
- relationship：双方明确确认的关系约定或重要共同经历。
- project：长期项目目标、架构方向与已确认决策。
- pending：明确尚未完成、以后需要继续的事项。
- correction：用户对旧事实的明确纠正。
- legacy：没有类型元数据的旧百炼节点，作为平滑兼容层。

### ContextPlanner

规划器完全在本地执行，不增加模型调用和主回复延迟。纠正、回忆、偏好、项目、情绪承接、短句续聊和一般聊天分别拥有不同的云端长期记忆开关、本地情节候选数、允许类型、检索模式和 ContextPack 分区预算。情绪承接不检索历史；短句续聊只允许最多一个本地近期情节兜底且需通过相关性门槛；项目与明确回忆才扩大相关历史。

### ConversationEval

- `run_context_engineering_eval.py`：完全离线检查查询路由、类型选择、当前输入保留和 token 硬上限。
- `run_conversation_quality_eval.py`：用脱敏场景做真实模型回放和裁判评分，覆盖上下文、记忆、自然度、连续性、重复、情绪对齐与篇幅。
- `analyze_runtime_log.py`：从 Trace 和组件日志统计端到端延迟、模型耗时、分段气泡、规划分布、token 使用率、记忆治理和错误。

## 后续可选阶段

- 用真实脱敏聊天扩充指代、临时状态覆盖和Critic误触发回归集。
- 为长链项目对话增加基于近期状态与摘要的检索查询改写，而不是直接使用“继续吧”一类低信息原话检索。
- 将动态内在状态以同样的短时、低权威方式提供给主回复，使主动消息和被动聊天的情绪连续性一致。

## 不采用的机制

- 不把完整聊天逐轮写入云端语义长期记忆，也不把全部历史塞进提示词；完整原文只进入本地归档，召回必须选择性进行。
- 不让助手回复成为用户事实证据。
- 不把所有历史一次性塞进提示词。
- 不为普通闲聊默认运行规划者、执行者、评论者三套模型。
- 不用静态关键词决定复杂语义或关系状态。
